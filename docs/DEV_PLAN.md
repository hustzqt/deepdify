---

## 审计摘要

项目不是从零开始。已有完整的认证、品牌 CRUD、AI 分析 API 骨架。  
Phase 1.5 的重点是：**修复 → 连通 → 补缺 → 交付 MVP。**

---

## Phase 1.5：MVP 交付（10 个工作日）

### Sprint 1：稳固地基（Day 01-03）

目标：所有已有功能可验证通过，零 P0 问题

| Day | 主题 | 具体任务 | 产出 | DoD |
|-----|------|---------|------|-----|
| 01 | 审计 + P0 修复 | ①项目结构审计 ②修复 register 路由重复 ③固定端口 ④统一 .env 键名 ⑤创建 .env.example | 审计报告 + 修复提交 | `pnpm dev` 在固定端口启动；无结构性路由冲突 |
| 02 | 现有功能回归 | ①登录/注册全流程 ②品牌 CRUD 全流程 ③用户资料编辑 ④AI 分析 API 调用 | 回归测试清单 | 所有现有页面可访问，API 返回正确 |
| 03 | Dify 连通验证 | ①Dify Cloud 注册+工作流创建 ②验证 `/api/dify` 端点 ③验证 `/api/ai/brand-analyze` ④确认 `DIFY_BASE_URL` 和 KEY 有效 | Dify 工作流截图 + API 响应日志 | 前端调用 AI 分析，返回结构化结果 |

### Sprint 2：AI 核心能力（Day 04-07）

目标：AI 品牌分析完整可用，聊天界面上线

| Day | 主题 | 具体任务 | 产出 | DoD |
|-----|------|---------|------|-----|
| 04 | AI 分析结果展示 | ①`BrandAiAnalyzePanel` 组件完善 ②加载状态/错误处理 ③分析结果持久化（存 DB） | 分析面板完整交互 | 提交品牌→看到 AI 分析→刷新仍在 |
| 05 | 聊天界面骨架 | ①`/app/chat/page.tsx` ②`ChatMessage` 组件 ③`ChatInput` 组件 ④聊天 Zustand store | 聊天页面可访问 | 能输入消息、看到消息列表（先 mock） |
| 06 | 聊天对接 Dify | ①Dify 聊天工作流 ②`/api/chat/route.ts` ③SSE 流式响应 ④前端流式渲染 | 真实 AI 对话 | 发消息→逐字显示 AI 回复 |
| 07 | 聊天历史 + 多轮 | ①`conversation_id` 管理 ②聊天记录存 DB ③历史会话列表 ④切换会话 | 完整聊天体验 | 关闭重开→历史还在 |

### Sprint 3：体验打磨 + 交付（Day 08-10）

目标：MVP 可演示，可部署

| Day | 主题 | 具体任务 | 产出 | DoD |
|-----|------|---------|------|-----|
| 08 | Dashboard 整合 | ①Dashboard 数据聚合 ②品牌卡片展示 ③最近分析展示 ④快速操作入口 | 完整仪表盘 | 登录后看到有意义的数据 |
| 09 | 全局体验优化 | ①响应式适配 ②暗色模式（如已有基础）③Toast 通知统一 ④加载骨架屏 | 体验一致性 | 移动端可用，交互流畅 |
| 10 | 部署 + 结案 | ①阿里云 ECS 部署 ②域名配置 deepdify.com ③HTTPS ④Phase 1.5 结案报告 | 线上可访问 | deepdify.com 可注册登录使用 |
