# 模型路由规则（Model Routing）

> 适用于：**IDE/Agent 辅助开发**、**内部自动化脚本**、以及 **Dify Workflow 条件节点** 的参考；线上用户请求仍以 **Dify CE + 宪法成本模型** 为准，不在此文件重复计费细则。

**与宪法对齐**：产品侧逻辑别名以 **`CONSTITUTION.md` §2.3 与附录B** 为准。**标准档为双轨**：中文/国内电商主路径 **`kimi-2.5` / `moonshot-kimi`**，英文/国际化/工具链并行 **`gpt-4o-mini`**，由 Dify **条件分支 + 监控** 选用或互为备份，**非二选一**。关键路径 **`gpt-5.3-codex`**。下表中的 `kimi2.5`、`deepseek3.2` 等为历史/示例写法，迁移时统一改为附录别名。

## 1. 策略评估（是否适用）

| 维度 | 结论 |
|------|------|
| **cheap-first** | 适用：大量探索、补全、简单重构可显著降本。 |
| **tiered escalation** | 适用：与「失败再升级」一致，避免一上来用大模型。 |
| **失败触发升级** | 适用：应用超时、解析失败、测试失败、置信度低作为客观信号，比主观判断稳定。 |
| **成本护栏** | 必须：`max_tokens`、任务预算、升级次数上限可防止滥用与雪崩。 |

**注意（非「唯一最优」）：**

- **模型名称与版本**（如 `kimi2.5`、`deepseek3.2`、`gpt-5.3-codex`）应以你们**当前供应商控制台实际可用 ID** 为准；本文件用**逻辑别名**，部署时映射到真实 `model` 字段。
- **任务分类**若过细，维护成本高；建议 **3～4 档**（cheap / standard / retry / critical）即可。
- **「置信度低」**需可观测指标（如自洽打分、多次采样方差、规则校验），否则易误判；可与「结构化校验失败」合并为一条 **quality_gate_failed**。
- **发布/迁移/支付** 除了模型档位，更依赖 **人工审批 + CI + 灰度**；模型升级是补充，不是替代。

## 2. 任务分类（路由输入）

| 档位 | 说明 | 默认路由（逻辑别名，与附录B一致） |
|------|------|----------------------|
| **T0 — cheap** | 注释、命名、简单格式化、文档草稿、重复劳动 | `deepseek-default` → 必要时再上探 |
| **T1 — standard** | 单文件小改、简单 CRUD、与现有模式一致；**中文为主** | **`kimi-2.5` / `moonshot-kimi`** 为主；**英文/工具链** 并行 **`gpt-4o-mini`**（非二选一，按语言/场景切换） |
| **T2 — retry / complex** | 超时/失败后重试、跨文件推理、疑难 bug、需强遵循规范 | `gpt-5.3-codex` |
| **T3 — critical** | 发布 checklist、DB 迁移、支付/权限/安全、核心业务不变量 | `gpt-5.3-codex`（或团队实测最稳模型） |

**产品与 Dify**：用户侧标准档路由以 **Workflow 条件节点**（如 `locale`、业务线）+ **监控**（延迟、错误率、结构化失败率）为准；**预算与 SLA 须在 A/B 实测后**写入第六章。

## 3. Fallback 链（每档）

- **T0**：`deepseek-default` → 失败再升级（护栏内）。
- **T1**：**中文/电商** → `kimi-2.5`；**英文/国际化** → `gpt-4o-mini`；失败可切换互为备份或升至 **T2**（未超 2 跳）。
- **T2 / T3**：默认 `gpt-5.3-codex`；若供应商限流，可配置同级备用（团队实测列表）。

## 4. 失败触发条件（自动升级）

满足 **任一** 即视为失败，可触发升级（在护栏内）：

1. **超时**（latency / provider timeout）。
2. **结构化输出校验失败**（JSON Schema / Zod / 约定字段缺失）。
3. **单测失败**（`vitest` / CI 步骤失败，且变更可归因于本次生成）。
4. **质量门失败**（`quality_gate_failed`：自洽检查、禁止模式、diff 过大等可编程规则）。
5. **（可选）置信度低**：仅当实现为可量化分数且阈值已校准。

## 5. 成本护栏（硬上限）

| 护栏项 | 建议 |
|--------|------|
| **每请求 `max_tokens`** | 按档位设上限（T0 最低，T3 可放宽但仍设 cap）。 |
| **每任务 `max_cost`** | 货币或 token 折算；任务结束或放弃时结算。 |
| **每次任务最多升级** | **2 跳**（cheap → mid → strong）；达到上限仍失败则 **人工介入** 或 **缩小任务范围**。 |
| **每日/每用户预算** | 与宪法「日预算」一致时，优先在 **服务端/Dify** 执行；IDE 侧另设软限制。 |

## 6. 与 BrandCraft / Dify 的关系

- **Next.js 应用内用户可见的 AI 调用**：仍走 **Backend-for-AI（如 `/api/dify`）**，由 **Dify Workflow** 选模型与知识库；本文件可作为 **Workflow 条件节点** 的对照表。
- **Cursor / 本地 Agent**：可直接按上表选模型；**不得**把高成本模型密钥暴露到浏览器。

## 7. 修订

- 模型别名与供应商变更时，只更新 **映射表**（环境变量或 Dify 控制台），不必改业务代码。
- Phase 结束或定价变更时，复审 **T0/T1 阈值** 与 **max_cost**。
